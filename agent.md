# 모두가 작가 - Multi-Agent Team 시스템 설계

## 시스템 개요

시각장애인이 음성만으로 책을 쓰고 출판할 수 있도록 돕는 멀티 에이전트 팀.
18개의 전문 에이전트가 협업, 릴레이, 검토, 개선, 피드백, 반복을 통해 최종 결과물(책)을 완성한다.

> **미션**: "말하면 글이 되고, 글이 책이 되어, 누구나 작가가 된다"

## A0. Orchestrator Agent (오케스트레이터)

### 역할
전체 에이전트 팀의 **총괄 지휘자**. 작업을 분배하고, 에이전트 간 릴레이를 조율하며, 워크플로우 패턴을 상황에 따라 결정한다.

### 핵심 권한
- 모든 에이전트에 작업 지시 및 우선순위 할당
- 워크플로우 패턴 결정 (순차/병렬/라우터/계층/반복)
- 에이전트 간 충돌 해결 및 최종 의사결정
- 진행 상태 모니터링 및 병목 탐지

### 워크플로우 패턴 결정 규칙

```
IF 작업이 단계적 의존성을 가짐:
    → 순차형 (Sequential)
    예: 기획 → UI/UX → 개발 → 테스트

IF 작업이 서로 독립적:
    → 병렬형 (Parallel)
    예: Frontend + Backend + STT 동시 개발

IF 작업 유형에 따라 다른 에이전트가 필요:
    → 라우터형 (Router)
    예: 사용자 요청 → "표지 관련" → A9.디자인 에이전트로 라우팅

IF 상위 에이전트가 하위 에이전트를 관리:
    → 계층형 (Hierarchical)
    예: A0 → A15(PM) → A3(FE) + A4(BE)

IF 결과물이 품질 기준을 충족할 때까지 반복:
    → 반복형 (Iterative)
    예: AI 글 작성 → 편집 → 품질 검토 → 미달 시 재작성
```

### 릴레이 프로토콜

```yaml
relay_message:
  from: "에이전트 ID"
  to: "에이전트 ID"
  type: "handoff | review_request | feedback | escalation"
  priority: "critical | high | medium | low"
  payload:
    task_id: "고유 작업 ID"
    context: "이전 에이전트의 작업 결과 요약"
    artifacts: ["결과물 파일 경로 목록"]
    requirements: "다음 에이전트가 수행할 작업"
    constraints: "제약 조건"
  feedback_loop:
    max_iterations: 3
    quality_threshold: 0.8
    escalate_on_failure: true
```

### A1. Service Planning Agent (서비스 기획 에이전트)

**역할**: 서비스 전체 방향, 사용자 요구사항, 기능 정의를 총괄

**담당 업무**:
- 시각장애인 사용자 페르소나 정의 (전맹/저시력/중도실명 등)
- 사용자 여정 맵 (User Journey Map) 작성
- 기능 정의서 (PRD) 및 우선순위 매트릭스
- 비즈니스 모델 캔버스 작성
- 경쟁 서비스 분석

**입력**: 프로젝트 요구사항, 시장 조사 데이터
**출력**: PRD 문서, 페르소나 시트, 사용자 여정 맵, 기능 우선순위 목록

**협업 릴레이**:
```
A1 ──출력: PRD──→ A2 (UI/UX 설계)
A1 ──출력: 기능 정의──→ A3 (Frontend), A4 (Backend)
A1 ──출력: 법적 요구사항──→ A13 (법률)
A1 ←──입력: 사용자 피드백──── A18 (사용자 대변인)
A1 ←──입력: 기술 제약──── A4 (Backend), A5 (STT)
```

**워크플로우**: 순차형 (가장 먼저 실행, 다른 에이전트의 기반)

---

### A2. UI/UX Design Agent (UI/UX 설계 에이전트)

**역할**: 접근성 중심의 음성 기반 사용자 경험 설계

**담당 업무**:
- Voice-First 인터랙션 설계
- 스크린 리더 탐색 흐름 정의
- 와이어프레임/프로토타입 설계
- 청각/촉각 피드백 체계 설계
- 화면별 ARIA 라벨 매핑

**입력**: A1의 PRD, 페르소나
**출력**: 와이어프레임, 인터랙션 명세서, ARIA 라벨 가이드, 음성 UX 시나리오

**협업 릴레이**:
```
A1 ──PRD──→ A2
A2 ──와이어프레임──→ A3 (Frontend 구현)
A2 ──음성 UX 시나리오──→ A5 (STT), A6 (TTS)
A2 ←──접근성 감사 결과──── A17 (접근성 감사)
A2 ←──사용자 테스트 피드백──── A18 (사용자 대변인)
```

**워크플로우**: 순차형 (A1 이후) + 반복형 (접근성 감사 피드백 루프)

**반복 조건**:
```
REPEAT:
    A2 설계 → A17 접근성 감사 → 미달 항목 피드백
UNTIL:
    WCAG 2.1 AA 기준 100% 충족
MAX_ITERATIONS: 5
```

### A3. Frontend Agent (프론트엔드 에이전트)

**역할**: Next.js + TypeScript 기반 접근성 웹 프론트엔드 개발

**담당 업무**:
- Next.js App Router 기반 페이지/컴포넌트 개발
- TypeScript 인터페이스 및 타입 정의
- WAI-ARIA 적용, 키보드 내비게이션 구현
- STT/TTS 클라이언트 연동
- 실시간 글 생성 UI (스트리밍 표시)
- 반응형 레이아웃 (데스크탑/모바일)

**입력**: A2의 와이어프레임, A4의 API 명세
**출력**: 프론트엔드 소스 코드, 컴포넌트 라이브러리

**협업 릴레이**:
```
A2 ──와이어프레임──→ A3
A4 ──API 명세──→ A3
A3 ←→ A4 (API 연동 테스트, 양방향 협업)
A3 ←→ A5 (STT 클라이언트 통합)
A3 ←→ A6 (TTS 재생 제어)
A3 ──빌드 결과──→ A12 (테스트)
A3 ←──접근성 감사──── A17
```

**워크플로우**: 병렬형 (A4와 동시 개발) + 반복형 (A12, A17 피드백 루프)

**코딩 규칙**:
```typescript
// 모든 인터랙티브 요소에 ARIA 필수
<button
  aria-label="녹음 시작"
  aria-pressed={isRecording}
  onKeyDown={handleKeyDown}  // Enter + Space 지원
  tabIndex={0}
>

// 모든 상태 변화에 aria-live 알림
<div aria-live="polite" aria-atomic="true">
  {statusMessage}
</div>
```

### A4. Backend Agent (백엔드 에이전트)

**역할**: FastAPI + Pydantic 기반 엄격한 타입의 백엔드 API 개발

**담당 업무**:
- FastAPI 라우팅, 미들웨어, 의존성 주입 설계
- Pydantic v2 Strict 타입 기반 전체 스키마 정의
- DB 모델링 (SQLAlchemy + Alembic 마이그레이션)
- WebSocket 엔드포인트 (STT 스트리밍, AI 글 생성 스트리밍)
- 인증/인가 (OAuth2 + JWT)
- API 버전 관리

**입력**: A1의 기능 정의, A2의 인터랙션 명세
**출력**: API 소스 코드, OpenAPI(Swagger) 문서, DB 스키마

**협업 릴레이**:
```
A1 ──기능 정의──→ A4
A4 ──API 명세(OpenAPI)──→ A3 (Frontend 연동)
A4 ←→ A5 (STT 서비스 프록시)
A4 ←→ A6 (TTS 서비스 프록시)
A4 ←→ A7 (AI 글 작성 API)
A4 ──코드──→ A11 (보안 감사)
A4 ──코드──→ A12 (테스트)
```

**워크플로우**: 병렬형 (A3와 동시 개발) + 계층형 (A5, A6, A7의 게이트웨이 역할)

**코딩 규칙**:
```python
from pydantic import BaseModel, Field, StrictStr, StrictInt, StrictFloat
from enum import Enum

class GenreType(str, Enum):
    ESSAY = "essay"
    NOVEL = "novel"
    POEM = "poem"
    AUTOBIOGRAPHY = "autobiography"

class CreateBookRequest(BaseModel):
    title: StrictStr = Field(..., min_length=1, max_length=200, description="책 제목")
    genre: GenreType = Field(..., description="장르")
    author_name: StrictStr = Field(..., min_length=1, max_length=100)
    chapter_count: StrictInt = Field(..., ge=1, le=100)

    model_config = {"strict": True}
```

### A5. STT Agent (음성인식 에이전트)

**역할**: 한국어 고인식률 STT 서비스 통합 및 음성 데이터 처리

**담당 업무**:
- STT 서비스 선정 및 연동 (CLOVA Speech 1순위, Whisper 2순위)
- 실시간 음성 스트리밍 처리 (WebSocket)
- 노이즈 제거, 음성 구간 감지 (VAD)
- 오디오 포맷 변환 (WebM → PCM/WAV)
- 인식 결과 후처리 (문장 분리, 구두점 삽입)
- 화자 분리 (다중 사용자 대응)

**입력**: 사용자 음성 스트림
**출력**: 정제된 텍스트, 타임스탬프, 신뢰도 점수

**협업 릴레이**:
```
사용자 음성 → A5 → 텍스트 → A7 (AI 글 작성)
A5 ←→ A3 (프론트엔드 음성 입력 UI)
A5 ←→ A4 (백엔드 WebSocket 엔드포인트)
A5 ──음성 데이터 정책──→ A11 (보안: 암호화/삭제 정책)
A5 ←→ A6 (음성 입출력 충돌 방지: STT 중 TTS 일시정지)
```

**워크플로우**: 병렬형 (A6와 독립 개발) + 반복형 (인식률 개선 루프)

**인식률 개선 반복 루프**:
```
REPEAT:
    A5 음성 인식 → A16 품질 측정 (인식률 테스트)
    IF 인식률 < 95%:
        파라미터 조정 (모델, 샘플링, 노이즈 처리)
UNTIL:
    한국어 인식률 >= 95%
MAX_ITERATIONS: 10
```

### A6. TTS Agent (음성합성 에이전트)

**역할**: 한국어 TTS 서비스 통합 및 음성 피드백 시스템 구축

**담당 업무**:
- TTS 서비스 연동 (CLOVA Voice, Google TTS 등)
- 시스템 안내 음성 vs 글 낭독 음성 분리
- 낭독 제어: 속도 조절, 일시정지, 문장 반복, 구간 이동
- 음성 큐잉 시스템 (안내 메시지 순서 관리)
- 사용자별 음성 선호도 저장 (속도, 목소리 타입)

**입력**: 텍스트 (시스템 메시지, AI 생성 글)
**출력**: 음성 오디오 스트림

**협업 릴레이**:
```
A7 ──생성된 글──→ A6 (낭독)
A8 ──편집 제안──→ A6 (수정 전/후 비교 낭독)
A6 ←→ A3 (프론트엔드 재생 컨트롤)
A6 ←→ A5 (STT/TTS 충돌 방지: 동시 사용 관리)
A6 ──음성 품질──→ A16 (품질 보증)
```

**워크플로우**: 병렬형 (A5와 동시 개발)

**음성 우선순위 큐**:
```
Priority 1 (즉시): 오류 알림, 긴급 안내
Priority 2 (높음): 사용자 명령 응답, 상태 변경 알림
Priority 3 (보통): 글 낭독, 편집 제안 읽기
Priority 4 (낮음): 배경 알림, 자동 저장 안내
```

### A7. AI Writing Agent (AI 글쓰기 에이전트)

**역할**: LLM을 활용한 문학적 글 생성 및 문체 관리

**담당 업무**:
- LLM API 연동 (Claude API 우선)
- 대화 텍스트 → 문학적 글 변환 프롬프트 엔지니어링
- 장르별 글쓰기 전략 (에세이/소설/시/자서전)
- 실시간 글 생성 스트리밍 (SSE)
- 작가 문체 학습 및 일관성 유지
- 챕터 구조 자동 생성

**입력**: A5의 정제된 텍스트 (사용자 음성 → 텍스트)
**출력**: 문학적으로 완성된 글, 챕터 구조

**협업 릴레이**:
```
A5 ──텍스트──→ A7 (글 작성)
A7 ──초고──→ A8 (편집/교열)
A7 ──생성된 글──→ A6 (TTS 낭독으로 사용자 확인)
A7 ←──편집 피드백──── A8 (수정 요청 시 재작성)
A7 ←──사용자 수정 지시──── A18 (사용자 대변인)
A7 ──장르/문체──→ A9 (책 디자인에 장르 정보 전달)
```

**워크플로우**: 순차형 (A5 이후) + 반복형 (A8과 편집 루프)

**글쓰기 반복 루프 (핵심 루프)**:
```
REPEAT:
    1. A5 → 사용자 음성을 텍스트로 변환
    2. A7 → 텍스트를 문학적 글로 변환
    3. A6 → 생성된 글을 사용자에게 낭독
    4. 사용자 → 음성으로 피드백 ("이 부분 수정해줘", "좋아 다음으로")
    5. IF 수정 요청:
           A7 → 해당 부분 재작성 → 3번으로
       ELIF 다음 진행:
           다음 챕터/섹션으로 이동
UNTIL:
    사용자가 "완성"이라고 선언
```

### A8. Editing Agent (편집/교열 에이전트)

**역할**: 4단계 편집 체계를 통한 원고 품질 관리

**담당 업무**:
- **구조 편집**: 전체 흐름, 챕터 구성, 논리 순서, 중복 검출
- **내용 편집**: 문장 다듬기, 문체 통일, 표현력 향상, 대안 문장 제시
- **교정**: 맞춤법, 띄어쓰기, 문법 (네이버 맞춤법 API, py-hanspell)
- **교열**: 최종 오탈자, 고유명사 일관성, 팩트 체크
- 원고 품질 리포트 생성
- 금칙어/민감 표현 감지

**입력**: A7의 초고
**출력**: 편집된 원고, 품질 리포트, 변경 이력

**협업 릴레이**:
```
A7 ──초고──→ A8
A8 ──편집 제안──→ A6 (TTS로 수정 전/후 비교 낭독)
A8 ──심각한 구조 문제──→ A7 (재작성 요청)
A8 ──편집 완료 원고──→ A9 (책 디자인)
A8 ──품질 리포트──→ A16 (품질 보증 최종 확인)
A8 ←──사용자 편집 지시──── A18 (사용자 대변인)
```

**워크플로우**: 순차형 (4단계 순서) + 반복형 (품질 미달 시 재편집)

**품질 리포트 구조**:
```yaml
quality_report:
  spelling_grammar:
    score: 0.0~1.0
    errors: [{location, type, suggestion}]
  style_consistency:
    score: 0.0~1.0
    issues: [{chapter, description}]
  structure:
    score: 0.0~1.0
    issues: [{type, description}]
  readability:
    score: 0.0~1.0
    grade: "초등~대학 수준"
  overall:
    score: 0.0~1.0
    verdict: "출판 가능 | 교정 필요 | 대폭 수정 필요"
    pass_threshold: 0.8
```

**4단계 편집 반복 루프**:
```
REPEAT:
    Stage 1: 구조 편집 → 구조 점수 확인
    Stage 2: 내용 편집 → 문체 점수 확인
    Stage 3: 교정 → 맞춤법/문법 점수 확인
    Stage 4: 교열 → 최종 점수 확인
    → 품질 리포트 생성
    IF overall_score < 0.8:
        미달 항목만 선택적 재편집
UNTIL:
    overall_score >= 0.8
MAX_ITERATIONS: 5
ESCALATE: A0 (오케스트레이터)에 보고
```

## 3개 감사/지원 에이전트 (Cross-Cutting)

이 3개 에이전트는 **모든 전문 에이전트를 가로질러** 작동하며, 최종 품질을 보장한다.

### A16. Quality Assurance Agent (품질 보증 에이전트)

**역할**: 전체 결과물의 품질 기준 충족 여부 최종 판정

**담당 업무**:
- 코드 품질 (린트, 정적 분석, 복잡도)
- API 응답 품질 (정확성, 일관성)
- STT 인식률 측정 및 벤치마크
- AI 글 품질 평가 (문학성, 일관성, 가독성)
- 원고 편집 품질 리포트 검증
- 책 디자인 품질 (조판, 인쇄 적합성)
- 최종 출판물 종합 검수

**품질 게이트 (Quality Gate)**:
```yaml
quality_gates:
  gate_1_code:
    lint_errors: 0
    type_errors: 0
    test_coverage: ">= 80%"
    blocker: true  # 미통과 시 다음 단계 진행 불가

  gate_2_stt:
    korean_accuracy: ">= 95%"
    latency_p95: "<= 500ms"
    blocker: true

  gate_3_writing:
    spelling_score: ">= 0.95"
    style_consistency: ">= 0.8"
    readability: ">= 0.8"
    blocker: true

  gate_4_book:
    layout_errors: 0
    image_resolution: ">= 300dpi"
    font_embedding: true
    blocker: true

  gate_5_accessibility:
    wcag_aa_compliance: true
    screen_reader_compatible: true
    keyboard_navigable: true
    blocker: true  # 가장 중요한 게이트
```

**협업 릴레이**:
```
모든 에이전트 ──결과물──→ A16 (품질 검사)
A16 ──불합격──→ 해당 에이전트 (재작업 요청)
A16 ──합격──→ A0 (다음 단계 진행 승인)
A16 ──품질 리포트──→ A15 (프로젝트 관리)
```

### A17. Accessibility Auditor Agent (접근성 감사 에이전트)

**역할**: 시각장애인 접근성의 최종 보증인. **이 프로젝트에서 가장 중요한 감사 에이전트.**

**담당 업무**:
- WCAG 2.1 AA 전체 항목 체크리스트 관리
- WAI-ARIA 적용 검증 (모든 인터랙티브 요소)
- 스크린 리더 호환성 테스트 시나리오 작성
- 키보드 전용 탐색 경로 검증
- 음성 피드백 완전성 검증 (모든 상태 변화에 음성 안내 존재)
- EPUB 접근성 태그 검증
- 색상 대비 비율 (저시력 사용자)

### A18. User Advocate Agent (사용자 대변인 에이전트)

**역할**: 시각장애인 사용자의 관점을 대변하고, 모든 의사결정에 사용자 중심 관점 반영

**담당 업무**:
- 사용자 시나리오 시뮬레이션 (전맹/저시력 사용자로서 전체 흐름 체험)
- 사용자 피드백 수집 및 전달
- 모든 에이전트 결과물에 대한 사용자 관점 리뷰
- 사용성 문제 사전 발견 및 보고
- 사용자 만족도 측정 기준 정의
- 실사용자 테스트 계획 수립

**사용자 시뮬레이션 시나리오**:
```
시나리오 1: 전맹 사용자 - 첫 회원가입
  "화면을 전혀 볼 수 없는 상태에서, 음성 안내만으로
   회원가입을 완료할 수 있는가?"

시나리오 2: 전맹 사용자 - 첫 글쓰기
  "음성으로 이야기를 들려주고, AI가 글로 변환한 결과를
   TTS로 듣고 수정 지시를 할 수 있는가?"

시나리오 3: 저시력 사용자 - 책 출판
  "완성된 원고의 표지를 선택하고, 출판 포맷을 결정하여
   전자책으로 발행할 수 있는가?"

시나리오 4: 중도실명 사용자 - 기존 글 편집
  "이전에 작성한 글을 불러와 특정 챕터를 수정하고
   다시 저장할 수 있는가?"
```

**협업 릴레이**:
```
A18 ──사용자 피드백──→ A1 (기획 반영)
A18 ──UX 개선 요청──→ A2 (UI/UX 수정)
A18 ──편집 지시 전달──→ A8 (편집: 사용자 의견 반영)
A18 ──사용성 문제──→ A0 (에스컬레이션)
모든 에이전트 ──결과물──→ A18 (사용자 관점 리뷰)
```

**워크플로우**: 상시 가동 + 반복형 (모든 결과물 리뷰)
